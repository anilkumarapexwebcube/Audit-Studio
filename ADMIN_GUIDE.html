Architecture overview

The system has three components working together:

ComponentWhere it runsJob
Master Google Sheet (gsc_last_crawl_Date) Google Sheets Holds the single source of truth: which Google account owns which domain, with access levels. The GSC_Config tab is the authoritative mapping. Auto-syncs weekly from GSC.
Apps Script Google's servers, owned by anilkumar.apexweb.cube@gmail.com Exposes the mapping data as JSON via a Web App URL. Also handles the crawl-date tracker on GitHub Pages (existing functionality, untouched).
Chrome Extension Each teammate's Chrome browser Reads the mapping from Apps Script. Each teammate signs into their assigned GSC accounts once. Generates PPTX reports by calling the GSC API directly with each user's OAuth tokens.

Why this design

  • The mapping lives in ONE place (the sheet). Update the sheet, the whole team sees the change instantly.
  • Teammates never see the sheet URL, the Apps Script URL, or the OAuth credentials.
  • Sign-ins are per-teammate, per-machine. Each person handles their own assigned client accounts.
  • Zero server hosting cost. No Railway, no VPS, no Docker.

First-time setup (already done, recap)

You have already done all of this:

  1. Google Cloud project with Search Console API enabled
  2. OAuth consent screen configured with ~75 test users (the GSC account emails)
  3. Web application OAuth Client ID created, with the extension's redirect URI authorised
  4. Apps Script Web App deployed with the get_config endpoint, set to "Anyone" access
  5. Extension folder configured with the Apps Script URL and OAuth Client ID pre-filled in config-default.js
  6. Standard install path established (C:\Tools\gsc-audit-studio) with that path's extension ID registered as the OAuth redirect URI
If you ever lose the OAuth credentials or need to re-create them, see the docs/SETUP.md file in the extension folder for the step-by-step Google Cloud Console walkthrough.

Distributing to teammates

Step 1: Edit config-default.js

Open the file in any text editor. Fill in the three values:

window.GSC_AUDIT_STUDIO_DEFAULTS = {
  appsScriptUrl: 'https://script.google.com/macros/s/.../exec',
  clientId:      '123456789-abcdef.apps.googleusercontent.com',
  clientSecret:  'GOCSPX-yourActualSecret'
};

These three values get baked into the extension. Teammates never have to enter them.

Step 2: Establish the standard install path

Chrome generates each extension's ID from a hash of the unpacked folder's location on disk. If two machines use the same path, they get the same ID. We'll use this to give everyone the same extension ID without needing to generate cryptographic keys.

The standard install path for the team is:

  • Windows: C:\Tools\gsc-audit-studio
  • Mac: /Users/Shared/gsc-audit-studio

Set this up on your own machine first:

  1. Extract the v1.2 folder to exactly C:\Tools\gsc-audit-studio
  2. Open chrome://extensions, enable Developer mode, click Load unpacked, select that folder
  3. Note the Extension ID that appears on the extension's card. It will be a 32-character lowercase string.
  4. Open Google Cloud Console → Credentials
  5. Click on your Web app OAuth client ID
  6. Under Authorized redirect URIs, add: https://YOUR_EXTENSION_ID.chromiumapp.org/ (with the trailing slash)
  7. Save the OAuth client

Every teammate who extracts the SAME zip to the SAME path (C:\Tools\gsc-audit-studio) will get the SAME extension ID, so the same redirect URI works for everyone.

What if a teammate uses a different path? Their extension ID will be different. Their OAuth sign-in will fail with "redirect_uri_mismatch". Two ways to fix:
  • Best: tell them to move their folder to the standard path and reload the extension
  • Acceptable: add their specific redirect URI to the Google Cloud OAuth client as a second authorised URI. You can have many URIs registered. This means each teammate with a non-standard path needs a one-off addition.
Why not use the cryptographic key approach? A pinned "key" field in manifest.json would give a stable extension ID regardless of folder path. But it requires OpenSSL, which is not pre-installed on Windows. The standard-path approach avoids that dependency and works just as well for an internal team where you can enforce the install location.

Step 3: Zip and share

  1. Right-click the extension folder → Send to → Compressed (zipped) folder (Windows) OR Compress (Mac)
  2. Rename the zip to gsc-audit-studio.zip
  3. Upload to a shared Google Drive folder visible to the team (not public)
  4. Share the link with teammates via Slack or email
  5. Attach the docs/TEAMMATE_GUIDE.html file so they have step-by-step install instructions
  6. In your message to the team, emphasize the exact install path: C:\Tools\gsc-audit-studio on Windows, no exceptions.
Security note: The zip contains your OAuth Client Secret. Anyone with the zip can extract it. Only share via private channels with people you trust. Never upload to public GitHub, Google Drive public links, or Slack channels accessible to non-team members.

Accessing admin mode

By default the extension shows only the Run Audit, Accounts, and Help tabs. The Domain List and Settings tabs are hidden from teammates.

To enable admin mode on your own machine:

  1. Open the extension dashboard
  2. Go to the Help tab
  3. Scroll to the bottom
  4. Click the "Admin mode" link in the small grey text
  5. When prompted, enter the admin code: admin123
  6. Domain List and Settings tabs appear. A gold "ADMIN MODE" badge shows in the top right corner.

To disable admin mode, click the same link again and confirm.

The admin code is hard-coded in the extension. To change it, edit dashboard.js (search for the string 'admin123') and re-zip. The current code is enough to keep curious teammates from poking around but is not a real security barrier — anyone determined enough can read the source. The real protection is that the Settings tab only modifies local-storage state; bad changes don't break anyone else's machine.

Adding a new client domain

You do nothing in the extension itself. The mapping lives in the sheet.

If the relevant Google account already has access to the new domain in GSC

  1. Open gsc_last_crawl_Date sheet
  2. Top menu: GSC Tools → Sync domains from GSC
  3. Wait 30 to 90 seconds. The script pulls every property from every authorised account.
  4. The new domain appears as a new row in GSC_Config automatically
  5. Tell the relevant teammate they can now generate a report for it

If the access was just granted and you need it ready in the next 5 minutes

Add the row manually to GSC_Config:

ColumnValue
Domainnewclient.com
GSC EmailThe full Gmail address that has access, e.g. searchconsoleanalyticsaccess8@gmail.com
Account KeyThe account-key shorthand (Gmail local part), e.g. searchconsoleanalyticsaccess8
Access LevelFull (URL prefix), Full (domain), or whatever applies
Last SyncedToday's date

Save the sheet. Tell the teammate to refresh the extension (it auto-fetches the mapping on open).

Adding a new Google account

This means a new GSC account (the Google account that owns properties in Search Console), not a teammate. For example a new searchconsoleanalyticsaccess9 account you just created for an agency.

  1. Add the new email to your Google Cloud OAuth consent screen test users list (OAuth consent screen). You have ~25 free slots remaining.
  2. Add a row to GSC_Config:
    Domain_setup
    GSC EmailThe new account's full email
    Account KeyThe Gmail local part
  3. In the sheet: GSC Tools → Generate auth URLs
  4. Open the Auth_Status tab. Find the new account's row.
  5. Click its auth URL. Sign into the new GSC account. Click Allow.
  6. GSC Tools → Check authorisation status to confirm it shows Authorised.
  7. GSC Tools → Sync domains from GSC to pull all properties from the new account into GSC_Config.

Adding a new teammate

  1. Send them:
    • The latest gsc-audit-studio.zip file
    • The TEAMMATE_GUIDE.html file (or a link to it)
    • A list of which Google accounts they will handle
  2. Add their personal Gmail (the email they use in their browser, not the GSC accounts) to your Google Cloud Console OAuth consent screen test users list. They cannot sign in to any GSC account through the extension until this is done.
  3. Tell them to follow the steps in the teammate guide
  4. Once they confirm setup is done, tell them which clients they own and which GSC accounts they need to connect
The test users list cap is 100 emails total. You have ~75 GSC accounts plus a few teammate personal emails, so you have headroom but not infinite. If you ever fill the list, you would need to publish the OAuth consent screen, which triggers Google's verification process (3 to 7 business days for the webmasters.readonly scope).

Removing a teammate (offboarding)

  1. Most important step: have them open chrome://extensions and click Remove on GSC Audit Studio. This wipes their local chrome.storage.local which contains the refresh tokens.
  2. For each GSC account they had connected: go to myaccount.google.com/permissions while signed in as that GSC account, find "GSC Audit Studio" in the apps list, and click Remove access. This invalidates their refresh tokens server-side even if they kept a copy of the extension folder.
  3. Remove their personal Gmail from the OAuth consent screen test users list (frees a slot)
  4. If they leave in bad faith or you suspect their machine is compromised, regenerate the OAuth Client Secret in Google Cloud Console → Credentials → your Web app client → Reset Secret. The old Secret stops working immediately. Then update config-default.js with the new Secret, re-zip, and distribute to the remaining team.

Updating the Apps Script

Routine: deploying a new version, same URL

This is what you do for normal code changes. The Web App URL stays the same, so the extension keeps working with no changes.

  1. Apps Script editor → make your code changes → Save (Ctrl+S)
  2. Top right → Deploy → Manage Deployments (not "New Deployment")
  3. Click the pencil icon (Edit) on the active deployment
  4. Version dropdown → New version
  5. Description: brief note about what changed
  6. Click Deploy

Special case: brand new deployment with a different URL

Only needed if the old deployment is broken or you want a fully clean slate.

  1. Apps Script editor → Deploy → New Deployment
  2. Type: Web app. Execute as: Me. Who has access: Anyone. Click Deploy.
  3. Copy the new URL
  4. Two options for getting it to the team:
    • Quick: tell teammates "open extension Settings tab (admin mode), paste new URL, save". Their existing extension folders keep working.
    • Clean: update config-default.js with the new URL, re-zip, distribute. Teammates unzip over their old folder and reload.
Always verify the new URL works BEFORE telling the team. Open it in an incognito tab with ?action=get_config appended. Confirm you see JSON with all your domains. If you see HTML or a login page, the "Who has access" setting is wrong.

Updating the extension code

Making changes

  1. Make code changes on your master copy of the folder
  2. Bump the version field in manifest.json (e.g. 1.2.0 → 1.3.0)
  3. Test on your own machine by reloading the extension on chrome://extensions
  4. If everything works, re-zip the folder
  5. Share with the team via Slack: "v1.3 released — replace your folder and click reload on chrome://extensions. Changes: [list]"

What teammates do

  1. Download new zip
  2. Unzip into the same location, overwriting old files (or delete old folder and unzip to same path)
  3. Open chrome://extensions and click the reload arrow on the GSC Audit Studio card

Their connected Google accounts, refresh tokens, and any custom settings (if they're in admin mode) all persist because they live in chrome.storage.local, which is keyed to the extension ID and survives folder replacement.

What if a teammate's extension ID changes

This should not happen if everyone uses the standard install path. But if it does (a teammate deletes their folder and re-extracts to a different path, or you change the standard path):

  1. Get the teammate's new extension ID from chrome://extensions
  2. Add https://NEW_ID.chromiumapp.org/ to your OAuth client's Authorized redirect URIs in Google Cloud Console
  3. They retry the sign-in. Should work immediately. No code changes needed.

Or, simpler: ask the teammate to move their folder to the standard path and reload the extension. The original redirect URI then works for them again.

Diagnostic checklist

SymptomLikely causeFix
Test connection in Settings returns "HTTP 404" Apps Script deployment URL has /u/N/ in the redirect chain (multi-account context) Redeploy Apps Script from an Incognito window signed in only as the owner. See "Updating the Apps Script" above.
Test connection returns "Connection OK" with HTML content instead of JSON Apps Script deployment "Who has access" is set to "Anyone with a Google account" or "Only myself" instead of "Anyone" Edit the deployment, change to "Anyone", redeploy
"OAuth Client ID not configured" on a teammate's machine config-default.js has empty clientId Edit config-default.js, re-zip, redistribute
Teammate's OAuth window says "App not verified" or "blocked" Either their personal email or the GSC email they're signing into is not in the test users list Add both to OAuth consent screen test users, wait 5 minutes, retry
"redirect_uri_mismatch" error in OAuth window on a teammate's machine Teammate installed the folder to a non-standard path, so Chrome generated a different extension ID Easiest fix: tell teammate to move folder to C:\Tools\gsc-audit-studio and reload. Alternative: ask teammate for their extension ID (visible on chrome://extensions), add https://THEIR_ID.chromiumapp.org/ as an additional redirect URI in the Google Cloud OAuth client.
Mapping is stale, missing recent domains Sheet sync hasn't run recently, or teammate's cache is stale Run "Sync domains from GSC" in the sheet menu. Tell teammate to enable admin mode and click "Refresh from Apps Script" in Domain List tab.
Reports for one specific domain fail with "No property found" Wrong account mapped in GSC_Config, or access was revoked Open the GSC account, verify the domain is still in the Search Console property list. If not, find the account that DOES have it and update the mapping row.
Manual Action / Security Issues screenshots are blank 5-second wait too short, or screenshot capture is failing on a specific machine Increase waitMs: 5000 to 8000 in dashboard.js, re-zip, redistribute. Or have teammate uncheck those slides for now.

How to inspect a failing report on a teammate's machine

  1. Ask them to open the dashboard, reproduce the failure
  2. Press F12 to open Chrome DevTools
  3. Click the Console tab
  4. Screenshot any red error lines
  5. Click the Network tab
  6. Look for failed requests (red status codes)
  7. Send both screenshots to you

For 90% of issues, the in-app log panel on the Run Audit tab is enough.
